# Technical appendix: PhysLab AI

Документ описывает архитектуру, данные, процедуру выбора моделей, итоговую оценку и границы применимости текущего MVP.

## 1. Архитектура

Проект разделён на общий слой и четыре предметных модуля.

```text
app.py
  ├─ core/
  │   ├─ upload_pipeline.py
  │   ├─ data_adapter.py
  │   ├─ recommendation_engine.py
  │   ├─ competency_engine.py
  │   └─ schemas.py
  ├─ labs/
  │   ├─ cooling/
  │   ├─ boyle_mariotte/
  │   ├─ isochoric/
  │   └─ heat_balance/
  ├─ data/
  ├─ models/
  ├─ evaluation/
  └─ tests/
```

Общий слой отвечает за загрузку, унификацию данных, схемы результата и интерфейс. Предметный модуль содержит генератор, извлечение признаков, модель, правила обратной связи и описание лабораторной работы.

## 2. Лабораторные задачи

### 2.1. Нагревание и охлаждение

Физическая основа: изменение температуры во времени и приближение закона охлаждения Ньютона.

Классы:

- `normal`;
- `single_outlier`;
- `sensor_drift`;
- `high_noise`.

Признаки описывают качество аппроксимации, остатки, локальные выбросы, дрейф и шум.

### 2.2. Закон Бойля — Мариотта

Физическая основа: постоянство произведения `pV` при неизменной температуре и количестве газа.

Классы:

- `normal`;
- `air_leak`;
- `temperature_change`;
- `volume_measurement_error`.

Генератор учитывает абсолютное давление, мёртвый объём, изменение температуры и потерю газа.

### 2.3. Изохорный процесс

Физическая основа: постоянство отношения `p/T` при неизменном объёме и количестве газа.

Классы:

- `normal`;
- `air_leak`;
- `volume_instability`;
- `temperature_sensor_lag`.

### 2.4. Тепловой баланс

Физическая основа: установление теплового равновесия с учётом теплоёмкости калориметра и потерь.

Классы:

- `normal`;
- `heat_loss`;
- `mass_measurement_error`;
- `insufficient_mixing`.

## 3. Генерация данных

Для каждой лабораторной работы формируется 160 экспериментальных семейств. Семейство задаёт общий контекст:

- физические начальные условия;
- профиль оборудования;
- параметры среды;
- длину и частоту измерений;
- базовый уровень шума.

Внутри семейства создаются четыре варианта, по одному на каждый класс. Такой подход не позволяет близким сериям с одинаковыми условиями попасть одновременно в обучение и итоговую проверку.

Итоговый объём на модуль — 640 серий.

## 4. Разбиение данных

Разбиение выполняется функцией `split_train_validation_test` из `labs/common/splitting.py`.

| Роль | Серии | Семейства |
|---|---:|---:|
| train | 320 | 80 |
| validation | 160 | 40 |
| test | 160 | 40 |

Стратегия: `stratified_group_train_validation_test`.

Инварианты:

- каждая строка относится ровно к одной части;
- все классы представлены в каждой части;
- генерационные семейства не пересекаются;
- `test` не используется для выбора модели или гиперпараметров.

Распределение сохраняется в `evaluation/<lab_id>/split_manifest.csv`. Параметры протокола — в `evaluation_protocol.json`.

## 5. Выбор модели

Базовый пайплайн сравнивает несколько семейств классификаторов на `validation`. Скрипт `validate_and_gridsearch.py` выполняет дополнительный подбор параметров с групповой кросс-валидацией на development-части.

Правило выбора:

- tuned-конфигурация принимается только при улучшении среднего `Macro F1` не менее чем на заданный порог;
- решение зависит от CV-результата, а не от `test`;
- без флага `--apply` рабочие модели не изменяются;
- после выбора модель переобучается на `train + validation`;
- итоговая метрика рассчитывается на сохранённой `test`-части.

## 6. Итоговые модели

Скрипт `evaluate_deployed_models.py` загружает фактически установленные `best_model.joblib`, применяет их к сохранённой тестовой части и создаёт единый отчёт.

| Лабораторная работа | Класс модели | Accuracy | Balanced Accuracy | Macro F1 |
|---|---|---:|---:|---:|
| Нагревание и охлаждение | RandomForestClassifier | 0,875 | 0,875 | 0,876 |
| Закон Бойля — Мариотта | RandomForestClassifier | 0,925 | 0,925 | 0,925 |
| Изохорный процесс | RandomForestClassifier | 0,994 | 0,994 | 0,994 |
| Тепловой баланс | GradientBoostingClassifier | 0,888 | 0,888 | 0,887 |

Для каждого модуля тестовая часть содержит 160 серий из 40 ранее не встречавшихся семейств.

Артефакты:

```text
evaluation/final_model_summary.csv
evaluation/<lab_id>/deployed_model_metrics.json
evaluation/<lab_id>/deployed_model_predictions.csv
```

## 7. Контроль надёжности

Файл `labs/common/reliability.py` отделяет классификацию известного сценария от случая вне области применения.

Профиль строится по `train + validation` и хранит:

- список признаков;
- устойчивые центральные значения;
- диапазоны;
- масштабы отклонений;
- служебные пороги.

Причины отказа:

- несколько признаков за допустимым диапазоном;
- сильное отклонение отдельного признака;
- низкая максимальная вероятность;
- малая разница между двумя ведущими классами.

При отказе результат содержит:

```text
predicted_class = unknown
candidate_class = <наиболее вероятный известный класс>
accepted = false
reliability_status = <причина>
```

Профили находятся в `models/<lab_id>/inference_profile.json`. Проверка проекта подтверждает, что список признаков совпадает с моделью и что профиль не использует `test`.

## 8. Реальные данные

Архивный аудит внешнего корпуса содержит 12 файлов и 25 таблиц. Он фиксирует форматы, столбцы, пропуски и физические величины. Первичные файлы не включены в публичную версию проекта.

Пайплайн нормализации поддерживает:

- CSV с разными разделителями и кодировками;
- листы Excel;
- сопоставление названий столбцов;
- преобразование единиц давления, объёма, температуры и времени;
- проверку пропусков, дубликатов и диапазонов;
- разбиение таблицы на отдельные серии;
- журнал принятия и отклонения физических каналов.

Строгий селектор температурных каналов принимает явные обозначения (`Temperature`, `Temp`, `T1`, `T_in`, `T_out`, `T_ambient`, `T_PCM`) и отклоняет давление, расход, влажность, массу, объём, ток, напряжение и время.

Поскольку `real_data_raw` отсутствует, сохранённые старые нормализованные файлы не считаются актуальным результатом исправленного селектора. Внешние прогнозы не участвуют в расчёте конкурсных метрик.

## 9. Признаки и интерпретация

Признаки рассчитываются по полной серии и имеют предметный смысл: отклонение от закона, наклон, форма остатка, устойчивость параметров, локальный шум, монотонность и другие характеристики конкретного опыта.

Пользовательский ответ включает:

- класс или `unknown`;
- вероятности известных классов;
- измеренные признаки;
- основания диагностической гипотезы;
- исследовательский вопрос;
- рекомендуемое действие;
- пометку о необходимости повторного опыта.

Вероятность модели не трактуется как вероятность истинности физического вывода.

## 10. Педагогический контур

Система использует подход teacher-in-the-loop:

- модель выполняет первичную диагностику;
- учащийся проверяет гипотезу и выбирает действие;
- преподаватель принимает педагогическое решение;
- итоговый вывод не генерируется вместо учащегося.

Профиль исследовательских действий строится по структурированным ответам. Он относится к конкретной работе и не является психологической диагностикой.

## 11. Использование инструментов ИИ при разработке

Инструменты генеративного ИИ применялись для чернового анализа требований, поиска вариантов реализации, ревью кода и подготовки тестовых случаев. Все изменения проверялись запуском кода, тестами и сопоставлением с физической моделью.

Примеры решений, принятых автором после проверки:

- отказ от псевдоразметки внешних серий;
- разделение `train`, `validation` и `test`;
- запрет выбора конфигурации по тестовой метрике;
- добавление содержательных экспериментальных семейств;
- отказ от обязательной классификации неизвестного входа;
- исключение нецелевых физических каналов при нормализации.

ИИ-инструменты не являются источником конкурсных метрик и не заменяют авторское понимание кода.

## 12. Инженерная проверка

Единая команда:

```bash
python quality_check.py
```

Этапы:

1. компиляция Python-файлов;
2. 53 автоматических теста;
3. проверка актуальности итоговых метрик;
4. 27 проверок структуры и артефактов.

`verify_project.py` проверяет:

- обязательные файлы;
- наличие наборов данных, моделей и отчётов;
- интерфейс `predict` и `predict_proba`;
- соответствие признаков модели и профиля надёжности;
- непересечение групп;
- актуальность итоговой оценки;
- 16 контрольных сценариев.

CI дополнительно выполняет Docker-сборку, запускает контейнер и проверяет Streamlit health endpoint.

## 13. Воспроизводимость

Установка:

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -r requirements.txt -r requirements-dev.txt
```

Проверка:

```bash
python quality_check.py
```

Базовое полное обучение:

```bash
python build_all.py --n-per-class 160
```

После полного обучения tuned-конфигурации при необходимости применяются отдельно:

```bash
python validate_and_gridsearch.py --mode quick --labs cooling boyle_mariotte --skip-streamlit --jobs 1 --apply
python validate_and_gridsearch.py --mode quick --labs isochoric heat_balance --skip-streamlit --jobs 1 --apply
python build_inference_profiles.py
python evaluate_deployed_models.py
```

## 15. Границы доказательств

Подтверждено:

- четыре модуля запускаются;
- модели различают сценарии генератора на новых семействах;
- тестовая часть не участвует в выборе конфигурации;
- установленные модели и отчёты согласованы;
- ненадёжный вход может быть отклонён;
- инженерные проверки воспроизводимы.

Не подтверждено:

- качество на размеченном корпусе реальных учащихся;
- переносимость на все марки оборудования;
- образовательный эффект;
- экономия времени преподавателя;
- качество групповой аналитики кабинета преподавателя.
